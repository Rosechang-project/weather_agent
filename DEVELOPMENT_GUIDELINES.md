# 🛠️ 專案開發規範與品質準則 (Development Guidelines)

本專案（Weather Agent）由執行長與開發團隊共同維護。為確保系統之穩定性、資安防禦能力及日後擴充之便利性，所有後續之代碼重構、功能開發與 AI 協作，必須嚴格遵守以下核心準則：

---

## 1. 永續性架構 (Sustainability)
* **嚴格禁止硬編碼 (Hardcoding)**：所有定位器、外部 API URL、存取金鑰（Tokens，台灣術語稱權杖）必須與業務邏輯徹底分離。
* **環境變數管理**：敏感資訊一律託管於本地端之 `.env` 檔案中，並透過 `.gitignore` 實施資安隔離。雲端生產環境（Production）則一律由 GitHub Secrets 加密注入。

## 2. 模組化與解耦設計 (Modular Design)
* **Page Object Model (POM)**：若涉及 Web 自動化或網頁端展示調整，必須遵循 POM 設計模式，確保元件與代碼具備高度可重複使用性與低耦合度。
* **功能解耦**：底層數據爬取（Low-level）與高階語意推理（LLM/Dify）需保持清晰邊界，避免交叉污染。

## 3. 品質導向與防禦性編程 (Quality & Stability)
* **異常處理 (Error Handling)**：所有網路請求、API 呼叫及資料解析區塊，必須建立完整的 `try-except` 異常捕捉機制。
* **日誌記錄 (Logging)**：系統關鍵節點（如連線成功、資料清洗完畢、推播失敗）必須包含清晰的日誌記錄，以利後續運維與 QA 偵錯。

## 4. 主流技術與通用工具 (Industry Standards)
* 專案內所採用之技術、套件與自動化排程工具（如 GitHub Actions、Playwright、Requests），必須以目前市面上企業廣泛使用之主流方案為主，避免引入冷門或非標準之第三方套件，以確保專案之技術含金量符合轉職市場需求。

## 5. 🤖 AI 協作行為限制與防禦 (AI Guardrails)
* **禁止非必要之大規模改動**：AI 在協助修正 Bug 或優化邏輯時，應採「最小更動原則」，僅針對有問題的代碼段落進行微調。嚴禁在未經執行長明確授權下，擅自重寫整支程式或大幅更動無關之邏輯鏈。
* **尊重並沿用既有架構**：嚴禁破壞或更動由資深工程師已封裝完成之底層函數（如 `build_final_ai_package()`）與既有資料結構。所有新功能必須在既有架構的框架下進行低耦合擴充。
* **嚴防 AI 幻覺 (Anti-Hallucination)**：AI 所提供之解決方案、套件方法（Methods）或語法，必須基於 2026 年當前穩定版本之官方文件。嚴禁憑空捏造不存在的套件參數、API 欄位或不合規之語法結構。

## 6. 溝通與用語規範 (Professionalism)
* 本專案之所有程式碼註解、README、日誌輸出及開發文件，一律使用台灣標準技術用語（例如：專案、權杖、資訊、變數、排程），嚴格禁止使用非台灣慣用之技術術語或浮誇字眼，以維持專案之專業度。
